### Full-data split and XGBoost grid search

The original working set (142k rows) excluded ~72k labeled galaxies that had images but no spectra — those rows were usable for tabular and image training but had been left out to keep the split consistent. Fixed this by generating a second split that keeps the test set identical (32,158 galaxies, fixed) and trains on all 214k labeled rows.

For XGBoost, ran a 27-combination grid search (n_estimators ∈ {100, 300, 500} × max_depth ∈ {4, 6, 8} × learning_rate ∈ {0.05, 0.1, 0.2}) with 5-fold stratified cross-validation on 154,855 training rows. Best configuration: 300 trees, depth 6, lr=0.2.

| Metric | Working Set (default) | Full Data + Grid Search |
|---|---|---|
| Macro F1 | 0.759 | 0.775 |
| Merger F1 | 0.396 | 0.439 |

The improvement in merger F1 (0.396 → 0.439) reflects both more training data and a better-tuned learning rate — with 154k examples the gradient signal is strong enough to support aggressive lr=0.2, which converges faster and avoids the over-regularisation of the conservative default (lr=0.05).

### ResNet-18 hyperparameter search

Tested four configurations on the full 214k training set, varying learning rate, dropout, and whether ImageNet pretraining was used:

| Config | LR | Dropout | Pretrained | Macro F1 |
|---|---|---|---|---|
| A (baseline) | 1e-4 | 0.0 | Yes | 0.795 |
| B | 5e-5 | 0.0 | Yes | 0.801 |
| C | 1e-4 | 0.3 | Yes | **0.807** |
| D | 1e-4 | 0.3 | No | 0.461 |

Config C (lr=1e-4, dropout=0.3) is the winner. Config D — training from scratch without ImageNet weights — collapses to macro F1=0.461, confirming that the pretrained feature representations are essential at this dataset scale. The dropout improvement (A vs C, +0.012) suggests the fully-connected head was overfitting without regularisation.

### Ablation study

Ran three additional fusion training runs, each zeroing out one modality during both training and evaluation (proper retrained ablation, not inference-time masking):

| Configuration | Macro F1 | vs Full Fusion | Merger F1 |
|---|---|---|---|
| Full fusion (all 3) | 0.873 | — | 0.668 |
| No image (spectral + tabular only) | 0.572 | −0.301 | 0.142 |
| No spectral (image + tabular only) | 0.852 | −0.021 | 0.641 |
| No tabular (image + spectral only) | 0.817 | −0.056 | 0.598 |

Image is by far the dominant modality: removing it drops macro F1 by 0.301 and merger F1 by 0.526. Without image, the model degrades to near spectral-only performance because tabular features alone cannot recover morphology. Tabular features contribute a meaningful +0.056 on top of image+spectral, mostly helping with borderline elliptical/spiral cases where photometric summary statistics (fracDeV_r, concentration) resolve ambiguity that the image model alone misses. Spectra contribute the least to morphological classification (+0.021), consistent with what we know: spectra encode physical type, not visual shape.

### Interpretability

**XGBoost feature importance.** Extracted gain-based importance from the tuned model. Top features by importance:

1. `fracDeV_r` (by far the highest — confirms it as the single strongest morphological indicator)
2. `color_g_r` (g-r colour index)
3. `concentration` (petroR90_r / petroR50_r)
4. `theta_0` (first eigenspectrum PCA coefficient)
5. `velDisp` (stellar velocity dispersion)

This validates the feature engineering choices made in Week 1–2 — the five features we expected to matter most are the five the model learned to use most. `fracDeV_r` ranking first is expected: it directly encodes whether a galaxy has a bulge-dominated or disc-dominated profile, which is the core distinction between ellipticals and spirals.

**GradCAM.** Applied Grad-weighted Class Activation Mapping on ResNet-18 Config C. Registered hooks on `layer4[-1]` (the deepest convolutional block), computed per-class activation maps for 2 examples per class. For ellipticals, activations concentrate on the bright central core; for spirals, activations spread along the disc; for mergers, activations highlight asymmetric features and tidal features at the edges. Generated without OpenCV (using PIL resize fallback), saved as 6-row × 2-column figure.

### Physical classification — stretch task

Re-labelled all 214k galaxies as star-forming (g-r < 0.65) vs passive (g-r ≥ 0.65), using g-r colour as a proxy for specific star formation rate (blue cloud vs red sequence). Trained both XGBoost (tabular) and 1D-CNN (spectral) on this binary task, using the same fixed test set.

| Model | Macro F1 | star_forming F1 | passive F1 |
|---|---|---|---|
| Tabular (XGBoost) | 0.9997 | 1.000 | 0.999 |
| Spectral (1D-CNN) | 0.935 | 0.946 | 0.924 |

The tabular result (0.9997) is near-perfect because `dered_g` and `dered_r` are raw inputs to the label formula — the XGBoost model trivially reconstructs g-r from its components. This is a methodological limitation, documented clearly in the code and the report: the tabular baseline has a direct numerical shortcut to the label that the spectral model does not have. Despite this, the spectral model achieves a genuine 0.935 — learning to infer star-formation status from Hα emission at 6563 Å, the Dn4000 break, and other spectral indicators without ever seeing the photometric magnitudes.

The contrast is striking: spectral macro F1 goes from 0.563 (morphological classification) to 0.935 (physical classification). This confirms the core hypothesis — spectra encode physical type, not visual shape. The 1D-CNN was the right tool all along, just for the wrong task.

## Challenges encountered

**`split_train_full.csv` missing `has_image`/`has_spectrum` columns.** The full-data split reads from `labeled_index.csv`, which doesn't carry image/spectrum availability flags. Fusion model's DataLoader tried to read `has_image` and crashed. Fixed by merging those columns from `working_set.csv` and filling non-working-set rows with `has_image=True, has_spectrum=False`.

**Physical classification label conflict.** The shared `evaluate()` function in `utils/metrics.py` is hardcoded for three-class morphological labels. Using it for the 2-class physical task produced a `ValueError` at classification report generation. Fixed by defining a local `evaluate()` and `print_results()` at the top of `physical.py` that use `PHYS_CLASSES = ["star_forming", "passive"]`.

**CUDA assertion for physical spectral model.** `SpectralDataset.__init__` maps labels via `LABEL_MAP["elliptical/spiral/merger"]` → 0/1/2. Physical labels are "star_forming/passive", so the map lookup failed with a `KeyError`. Created `PhysicalSpectralDataset` that reads `phys_label_id` (0/1) directly from the dataframe, bypassing `LABEL_MAP` entirely.
